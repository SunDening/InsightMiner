
KB 知识库描述长期优化方案
=========================

当前状态：方案 A（启动时 LLM 摘要，mtime 缓存）
  位置：build_kb_descriptions() → classify_intent_node 引用 _kb_descriptions


方案 B：ChromaDB 级动态感知
--------------------------
状态：未实施

思路：
  - 不从文件名生成摘要，改为从 ChromaDB collection 的 metadata 中提取文档来源
  - 这反映的是实际已索引的内容，与索引状态完全一致
  - 支持运行时上传文档后自动感知（因为上传后必定建索引）

实施时机：
  - 实现文档上传功能时一起做
  - 需要一个文档 metadata 持久化机制（文件名→文档名/描述 的映射表）


方案 C：缓存 + 失效通知
--------------------------
状态：未实施

思路：
  - 启动时提取一次摘要，缓存到全局变量
  - 文档变更时通过钩子/回调刷新缓存（如 FastAPI endpoint 触发的上传/删除事件）
  - 每次分类零额外开销

实施时机：
  - 做上传功能时，在上传/删除 API 中加一行缓存失效调用即可


n. 知识库中文档过多或文档中字数太多时，为每个文档生成摘要，把摘要向量化。
   在检索时，先通过匹配摘要找到证据可能存在的文档（尽可能多召回），
   之后临时向量化相应文档内容并做匹配。


知识库不同语言导致向量化后难以根据语义匹配。
如果要支持多语言，可能需要更强大的嵌入模型。
