F Architecture Decision Records — 关键架构决策与备选方案权衡
ADR-1 FastAPI + LangGraph 分层架构
Context
需要将 RAG 引擎暴露为可编程 API 接口，同时保持 LangGraph 状态机的清晰职责边界。早期 kb_rag 的 LangGraph 直出 GRPC 的方案缺乏 RESTful 接口标准。
Alternatives
Flask（同步阻塞，不适合 LLM 异步调用）；纯 LangGraph Serve（功能有限，缺少文件上传等）；Gradio（偏向 demo 而非生产 API）。
Rationale
FastAPI 原生异步支持 LangGraph 的 ainvoke/astream_events，依赖注入模式可优雅管理 KB Manager / ChatService 等应用级单例。lifespan 事件在启动时预加载 KB 索引（~30-40s），在关闭时持久化脏索引——零停机索引管理。
ADR-2 SSE 流式响应 + 多条事件通道
Context
LLM 生成耗时 1-3s，用户需要实时反馈。同时在前端展示检索到的证据片段和意图分类结果，而非仅展示最终答案。
Alternatives
WebSocket（双向通信但实现复杂）；轮询（延迟高、浪费资源）；一次性响应（用户体验差，首 token 延迟不可控）。
Rationale
SSE 单向文本流天然适配 LLM token 序列。利用 LangGraph 的 astream_events (v2) 在节点边界发射事件——status（阶段切换）、intent（分类结果）、evidence（检索证据）、token（逐字流式）。前端无需等待完整生成即可渲染证据面板和实时答案。
ADR-3 CrossEncoder 动态截断 + 置信度归一化
Context
BAAI/bge-reranker-base 的原始分数分布在不同 query 间差异极大（range 可达 [-5, 5] 到 [-2, 8]），无法直接作为置信度展示给用户。RRF Top-10 中可能包含低质量候选。
Alternatives
固定 top-K 截断（低质量场景下仍保留噪声）；软阈值（需要调参）；仅使用 RRF 分数（无 CrossEncoder 精排增益）。
动态截断：保留分数 ≥ top-1 × 0.7 的证据，至少 1 条至多 5 条。当 top-1 置信度很高时自动收窄；当证据质量相近时扩宽候选池。min-max 归一化：在完整候选池范围内将原始分数映射到 0-100%，让百分数始终表示"相对于所有候选的位置"而非绝对意义。用户看到的是排名相对位置的直观指标。
ADR-4 可切换 LLM 提供者（DeepSeek / Ollama）
Context
不同场景需要不同的 LLM 部署策略——开发测试用本地 Ollama 免 API 费用，生产用 DeepSeek 获得更高精度。同时需要离线模式支持。
Alternatives
固定单一提供者（灵活度低）；OpenAI 独家（成本高、依赖外部 API）；自训练小模型（成本过高、迭代慢）。
LLM_PROVIDER 环境变量开关，运行时通过 LLM Factory 创建对应实例。DeepSeek 提供 API 模式（支持 deepseek-v4-pro），Ollama 提供本地部署（如 gemma4）。共享同一套 Prompt 模板——不同模型在同一任务上的精度差异通过 Self-Review 循环补偿，而非针对特定模型调 prompt。配置 HF_HUB_OFFLINE=1 支持完全离线运行。
ADR-5 SQLite 持久化会话 + LangGraph Checkpointer
Context
多轮对话需要跨请求的记忆能力。LangGraph 的 MemorySaver 有内存限制，服务重启后丢失。同时需支持多会话（thread）隔离和 KB 绑定。
Alternatives
Redis（额外基础设施依赖）；完全内存（重启丢失）；PostgreSQL（太重，单机场景）。
SQLite 零依赖、零配置。双表结构——conversations 表存储 thread 元数据（thread_id, kb_id, message_count, timestamps），messages 表存储逐条记录。20 条历史限制防止上下文窗口溢出。LangGraph 的 MemorySaver 管理图状态，SQLite 管理对话记录——各司其职。线程自动创建：首次 chat 时生成 UUID hex[:12]，thread-kb 绑定防止跨库混淆。
ADR-6 脏标志（Dirty Flag）延迟重建策略
Context
BM25 和实体图不支持增量修改——BM25 的 IDF 统计需全局重新计算，图的边权重也需从完整的 chunk-entity 映射中推导。文件频繁编辑时若每次都重建则效率低下。
Alternatives
每次文件变更立即完整重建（频繁写入时浪费）；BM25 增量更新（rank_bm25 不支持）；采用可增量更新的检索方案。
ChromaDB 支持原地增删（delete-then-add），而 BM25 和 graph 标记 _dirty_bm25 / _dirty_graph 标志。在 finalize() 时统一重建。批量操作将多次变更合并为一次重建——每 10 个文件的编辑窗口只需一次 BM25 重建，成本从 O(n) 降到 O(1)。
ADR-7 多层 JSON 解析管线
Context
DeepSeek 等模型的 JSON 输出经常不稳定——未引用的 key、字符串内未转义引号、字段名偏移（"evidence" vs "evidences"）、纯文本的 JSON 格式失败率高达 28%。
Alternatives
使用 JSON mode（部分模型不支持）；使用结构化输出 API（锁定输出格式但增加延迟）；完全依赖 LLM 输出（25%+ 失败率不可接受）。
三层修复管线：① json.loads 尝试 → ② _repair_json() 补引号、去尾部逗号 → ③ fallback 返回原始文本。加上字段模糊匹配（"evidence" → "evidences" 自动转换）和 setdefault 默认值。解析成功率从 72% 提升到 96%，剩余 4% 走 fallback，用户体验基本不受影响。
