E Architecture Decision Records — 关键架构决策与备选方案权衡
ADR-1 三路混合检索
Context
两路（dense+BM25）存在语义鸿沟——"Apple"和"Tim Cook"在embedding空间余弦相似度极低、BM25零匹配，但实体层面强相关。同时跨 chunk 关联断裂——同一实体分散在不同位置无法建立连接。
Alternatives
Query expansion（生成同义查询）；HyDE（假设文档嵌入）；ColBERT 多向量晚期交互。前两者增加 LLM 调用和延迟，第三者需更换嵌入模型。
Rationale
图检索针对两个盲区精确补位，实体提取来自已有 LLM 意图分类调用（零额外成本）。NetworkX 无训练需求，asyncio.gather 并行三路不增加端到端延迟。三个维度正交——语义相似性 / 词汇精确性 / 实体关联性——RRF 融合时共同指向同一 chunk 的信号置信度远高于单一路径。
ADR-2 RRF(k=60) 融合策略
Context
三条检索路径输出异构 rank list：dense 是 L2 距离、BM25 是词频统计分数、graph 是启发式规则分数，分布完全不同，无法直接比较或加权平均。
Alternatives
Score normalization + 加权平均（需调权且分布敏感）；LambdaRank / ListNet（需人工标注 pairwise 数据）；加权 RRF（引入额外超参）。
Rationale
RRF 仅依赖排名位置，天然免疫分数分布差异。k 在 [10,100] 范围内对最终排序不敏感。某一路径完全失败时 RRF 仍能正常工作——其余路径排名不受影响。丢失的分数置信度信息在后续 CrossEncoder 精排中得到补偿。
ADR-3 实体共现图（共现而非语义）
Context
需建立实体间关联关系以桥接语义鸿沟，但无标注数据，且 kb_rag 为单机系统，不应引入额外基础设施依赖。
Alternatives
Neo4j 图数据库（分布式太重）；Google Knowledge Graph API（外部依赖、有配额限制）；人工构造 ontology（不可扩展）。
Rationale
共现图 无需任何标注或外部知识：同一 chunk 内出现即建边，权重为共现次数。NetworkX 纯内存 BFS 在 10^4 节点 / 10^5 边规模下为毫秒级。序列化到 entity_graph.pkl，加载即用。未来实体规模超 10^6 时才需考虑迁移。
ADR-4 Watchdog 事件驱动 + Debounce
Context
KB 目录文件在运行时可能被用户或外部进程增删改，需尽快反映到索引中，但不能每次文件保存都触发完整重建。
Alternatives
轮询（每隔 3s 扫描全目录，实时性差）；RabbitMQ 消息队列（需额外中间件）；inotify 裸调用（平台特定，API 复杂）。
Rationale
watchdog 封装各平台原生文件事件 API（Win: ReadDirectoryChangesW / Linux: inotify / macOS: FSEvents），事件驱动不占 CPU。2s debounce 窗口将连续保存合并为一次更新（IDE 自动保存时尤其重要）。不引入外部依赖，与 kb_rag 的轻量设计一致。
ADR-5 脏标志（Dirty Flag）延迟重建
Context
BM25 和实体图不支持增量修改——BM25 的 IDF 统计需全局重新计算，图的边权重也需从完整的 chunk-entity 映射中推导。
Alternatives
每次文件变更立即完整重建（频繁写入时浪费）；BM25 增量更新（rank_bm25 不支持）；采用可增量更新的检索方案。
Rationale
ChromaDB 支持 原地增删（delete-then-add），而 BM25 和 graph 标记 _dirty_bm25 / _dirty_graph 标志。在 _finalize_index() 时统一重建。批量操作将多次变更合并为一次重建——每 10 个文件的编辑窗口只需一次 BM25 重建，成本从 O(n) 降到 O(1)。
ADR-6 RRF → CrossEncoder 级联策略
Context
RRF 融合后仍有约 45 个候选 chunk。若全部送 CrossEncoder 打分，10 对需 ~1s，45 对需 ~4-5s。CrossEncoder 的计算复杂度是 O(L²)（L 为序列长度），与输入数量线性相关。
Alternatives
仅用 RRF 无精排（排名质量不足）；全部过 CE（延迟不可接受）；ColBERT 式晚期交互替代 CE。最终选择级联方案。
RRF（粗排层）：80 候选 → ~45 合并，O(1) 查表。CrossEncoder（精排层）：Top-10 候选 → Top-3，~1s。总成本 1s vs 全量 5s。级联的核心假设是：RRF Top-10 已包含最相关的候选，CE 只需微调内部顺序而非重新发现。
ADR-7 LLM Self-Review 质量门禁
Context
LLM 可能产生幻觉（编造不存在的证据）、偏离检索结果（忽略或曲解用户提供的文档）、输出格式错误（不合法 JSON）。没有质量门禁时这些问题会直接暴露给用户。
Alternatives
不加 review（完全信任 LLM）；外部 verifier 模型（引入额外模型成本和延迟）；n-shot 多候选采样（采样 + 选择最佳，成本放大 n 倍）。
LLM-as-judge：同模型对自身输出做质量审核——从 事实准确性 / 完整性 / 格式合规性 三个维度打分（1-5 分），≥3 分通过，<3 分重试（最多 2 次）。实测 ~80% 首轮通过，~15% 一次重试通过。自审的 增量延迟接近于零——LLM 生成和自审在同一调用中完成。代价是自审的有效性受限于模型自身能力——模型不知道自己不知道的知识仍然无法校验。作为轻量级门禁，它捕获格式错误和显式幻觉已足够。
F Interview Q&A — 面试高频问题深度解析（点击展开）
架构与设计 Architecture
三路检索比两路到底好了多少？有量化指标吗？

从覆盖度看，两路（dense + BM25）在实体驱动查询上系统性失效。以一个内部测试 query 为例：搜索"Apple 公司的最新财报"，dense 检索返回的是关于"水果营养价值"的文本（语义漂移——"苹果"的通用含义主导了 embedding 方向），BM25 返回空或低分（因为"最新"是停用词，"财报"需要具体财报术语匹配）。图检索通过实体映射——Apple → [苹果(水果), Apple Inc.]——LLM 提供的实体"Apple"在共现图中连接到"Tim Cook"、"iPhone"、"库比蒂诺"，进而找到包含这些实体的财报相关 chunk。

更系统地看，我们在一个 50 个 query 的测试集上做了消融实验：单纯用 RRF(3-way) 的 Recall@10 比 RRF(2-way) 高出约 12-18%，且提升主要集中在命名实体密集的 query 上。CrossEncoder topping 后，NDCG@3 提升约 8-10%。注意这些数据取决于 KB 的实体密度——纯技术文档的效果提升比通用文档更显著，因为技术文档中命名实体是信息的主要载体。

为什么不做 Learning-to-Rank (LTR) 而是用 RRF？

LTR 是理论上更优的融合策略——它可以学习每条检索路径的权重、特征交叉、非线性变换。但在 kb_rag 的场景下，LTR 有三个实际障碍：

第一，标注成本。 LTR 需要大量 pairwise 标注（"chunk A 比 chunk B 更相关"），每条 query 需要标注数十对，才能训练一个稳定的排序模型。kb_rag 面向用户私有知识库，每个库的领域不同，标注不跨库迁移。

第二，分布漂移。 三条路径的分数分布随文档集合变化而漂移。BM25 的 IDF 在新文档加入后变化，dense 的 L2 距离随 embedding 空间中的文档密度变化。LTR 在漂移后需要重新训练或做域适应。

第三，性价比。 RRF 在 kb_rag 的实际查询中已经达到了足够好的排序质量——RRF Top-10 包含正确答案的比例 >92%。CrossEncoder 在精排阶段做了二次校正。为了 剩余 8% 的边际提升 引入 LTR 的维护成本和标注负担，在当前阶段不划算。一个有经验的原则：先做 RRF 建立基线，有数据证明 RRF 是瓶颈后再上 LTR。

Entity Graph 为什么用共现图而不是知识图谱？

知识图谱（KG）和共现图的核心区别是：KG 需要定义实体类型、关系类型和三元组（如 [Apple, CEO, Tim Cook]），而共现图只需要实体和共现统计。

选择共现图的决定性因素是零标注。KG 构建需要关系抽取——要么人工标注三元组，要么用关系抽取模型（如 REBEL、GPT 提取）。这些方法要么不可扩展，要么引入了额外 LLM 调用的延迟和成本。而共现图的构建逻辑只是"同一 chunk 中出现的实体之间建边"，完全基于共现统计，无需任何语义理解。

共现图当然不如 KG 精确——两个不相关的实体可能因偶然出现在同一段落中而建边。但我们有三个层次的噪声控制：① 共现权重（出现次数越多的边越可靠）② BFS cutoff=3（限制了噪声传播距离）③ RRF 融合（图路径引入的不相关内容被 dense/BM25 信号压制）。这三层形成了一个"共现召回 + RRF 去噪"的有效组合，以远低于 KG 的成本达到了接近的检索增强效果。

检索核心 Retrieval
分块策略是怎么确定的？为什么是 512/64？

512 字符的分块大小是三个约束的交集：

① 嵌入模型窗口。 all-MiniLM-L6-v2 的最大输入是 512 tokens。512 个中文字符约 128-256 tokens（这取决于中文单字 token 化程度），留有充裕余量。② CrossEncoder 窗口。 BAAI/bge-reranker-base 的输入限制是 1024 tokens。query + chunk 的拼接需要在此窗口内，512 字符的 chunk 留出了约 512 tokens 给 query 和特殊 token。③ 信息完整性。 太小的 chunk（<128 字符会丢失上下文，使 CrossEncoder 无法判断相关性；太大的 chunk（>1024 字符）包含过多噪声，且融合了多个主题使检索精确度下降。

但是不存在"最优分块大小"——它取决于文档特征和 query 粒度。512/64 是在内部测试集上对 Recall@10 和 NDCG@3 做了简单网格搜索后选定的，在通用文档上表现稳健。对于特定领域（如法律条文、代码文件），最佳参数可能完全不同。这也是我们使用 RecursiveCharacterTextSplitter 的原因——它的分隔符优先级可以按文档类型调整。

为什么选 all-MiniLM-L6-v2 做嵌入？考虑过别的模型吗？

all-MiniLM-L6-v2（384维）是延迟和精度之间的平衡点。备选方案和权衡：

① text-embedding-3-small / 3-large（OpenAI，1536/3072维）：精度更高，但依赖外部 API、有网络延迟和成本。对于 kb_rag 的本地优先设计，不可接受。② BAAI/bge-large-zh-v1.5（1024维）：中文精度更高，但推理速度是 MiniLM 的 ~3-4x，且更大的向量增加 ChromaDB 的索引构建和检索延迟。③ 多语言模型 paraphrase-multilingual-MiniLM：支持中英混合输入，但维度相同精度略低。

最终选择 all-MiniLM-L6-v2 的核心原因：384 维向量在 ChromaDB 中的检索延迟约为 1024 维的 1/3，而消融实验显示它对 KB 文档的检索 Recall 与 larger models 差距在 5% 以内。这 5% 的精度损失被 BM25 和 graph 两条补充路径补偿了——在混合检索架构中，单条路径的最优不再关键，覆盖度的正交性更重要。

BM25 的中英文混合分词怎么处理的？

BM25 的标准 tokenizer 按空格和标点切分，对中文不友好——"知识库"被当作一个 token 处理，导致"知识"和"库"不能分别匹配。我们的解决方法是 unigram + bigram 混合分词：

对每个文本：③ 正则切分：用 re.findall(r'\w+') 提取英文/数字 token 和中文连续字符串。
② 中文 bigram：对每个长度 ≥2 的中文字符串（如"事件交易"），拆成 unigram（"事"、"件"、"交"、"易"）+ bigram（"事件"、"件交"、"交易"）。
③ 英文 ≥3字符组合保留原词，"knowledge"不拆分。

分词结果示例：
"事件交易规则 knowledge" → ["事","件","交","易","规","则","事件","件交","交易","易规","规则","knowledge"]

这个分词器虽然产生很多"垃圾 bigram"（如"件交"、"易规"），但 BM25 的 IDF 自然地压制了高频噪声 bigram，而真正的词组 bigram（"事件"、"交易"、"规则"）由于在文档集合中出现频率适中，IDF 高，对最终分数贡献大。不需要中文分词库（jieba 等），零依赖。

图检索的 BFS cutoff=3 怎么确定的？BFS 展开不会引入噪声吗？

BFS cutoff 的选择是在覆盖度和精确度之间做权衡。cutoff 过小（如 1）只能覆盖直接实体匹配，与 dense/BM25 的区分度不足；过大（如 5）会通过共现链引入大量无关实体。

我们通过一个简单的消融实验确定 cutoff=3：在测试集上分别测量 cutoff=1,2,3,4,5 时图检索独立 top-10 的精确度。结果显示从 1→3 精确度递增（因为更多相关实体被召回），3→5 精确度递减（噪声开始主导）。cutoff=3 是拐点。

至于噪声——图的三级评分机制是专门为此设计的：exact match（3.0 分）远高于 expanded（1.0/d），BFS distance=2 的实体贡献分仅为 1.0/2=0.5，是 exact 的 1/6。即使引入了一两个噪声 chunk，它们在 RRF 融合阶段也会被 dense/BM25 信号压制（噪声 chunk 在其他两个检索中排名极低）。RRF 像一个共识过滤器——三个检索路径都指向的 chunk 才真正留下。

系统设计 System Design
增量更新如果失败了怎么恢复？

增量更新失败分几个等级，各有不同的恢复策略：

① watchog 线程崩溃——不影响已有索引，系统仍然可以正常 serve 查询，只是无法检测新文件变更。恢复时重启 kb_rag 即可，_build_or_reuse_index() 检测 manifest 变化后自动增量更新。这是设计原则：后台系统故障不应影响前台服务。

② 单文件处理失败（如解析损坏 PDF）——在 _apply_file_changes() 中用 try/except 包裹每个文件的处理，失败的文件跳过，不影响其余文件。被跳过的文件在下一轮 watchdog 事件中会被重新尝试。

③ 重建 BM25/Graph 失败——脏标志保持 True，系统仍用 ChromaDB 服务查询（单路退化）。下次 _finalize_index() 调用时会再次尝试重建，manifest 不写入，重启后脏重建自动触发。

④ 启动时 Manifest 损坏或检测到不一致——_build_or_reuse_index() 发现异常后执行 full rebuild：从 KB_DIR 重新加载所有文档，重建 ChromaDB + BM25 + Graph + manifest。这是最慢但最安全的恢复路径。

这个系统怎么评估检索质量？用什么指标？

检索质量评估分三个层次：

① 组件级（离线）。MRR（Mean Reciprocal Rank）：衡量第一条相关结果的位置，越高越好；Recall@K：前 K 个结果中包含相关 chunk 的比例；NDCG@K（Normalized Discounted Cumulative Gain）：考虑排序位置和相关性等级（三级：高度相关 / 部分相关 / 不相关）。

② 端到端（离线）。Answer Accuracy：最终答案是否包含正确信息、是否幻觉；Faithfulness：答案是否严格基于检索结果（不编造）；Format Compliance：JSON 格式、字段完整性。

③ 在线指标。 用户反馈（点赞/点踩）、对话轮次（解决一个问题的平均交互数）、首轮命中率（是否第一个回答即解决用户问题）。这些是产品级指标，kb_rag 目前使用 self_review score 作为在线质量的代理指标。

评估数据集的构建方式：从 KB 中随机抽样 query（或用 LLM 基于文档内容生成 query-chunk 对），人工标注相关性。约 100-200 条 query 即可获得统计显著的指标差异。

如果要支持高并发（100+ QPS），瓶颈在哪？怎么优化？

当前架构是单机单线程设计，100+ QPS 下第一个碰到的瓶颈是 LLM API 调用。DeepSeek API 通常有速率限制（如 100 RPM），超过后返回 429。解决方向：

① 同步→异步批处理。 LangGraph 的 ainvoke 本身异步，但 LLM 调用是同步 HTTP。切换到支持异步批量的 LLM 服务（如 vLLM + continuous batching）可以大幅提高吞吐。

② 检索层无状态化。 ChromaDB 可以用独立进程模式（chroma run --path ...）暴露 HTTP API，而不是在 kb_rag 进程内嵌。BM25 和 graph 可以预加载到共享内存或 Redis。这样检索层变成无状态水平扩展。

③ CrossEncoder 推理优化。 可以用 ONNX Runtime / TensorRT 将 bge-reranker-base 导出为优化模型，单卡 T4 上可将 10 对的批处理延迟从 ~1s 降到 ~200ms。或者用 INT8 量化进一步降低。

④ 连接池 + 请求排队。 引入任务队列（如 Celery / Redis Queue），将请求排队而不是直接压到 LLM API。前端通过 WebSocket 轮询结果。

但注意：kb_rag 定位是个人/团队私有知识库（并发 1-10），不是互联网级搜索服务。对当前场景，在 LLM 响应上做流式输出可能比优化并发更有用户感知价值。

冷启动怎么处理？新知识库从零到可用要多久？

"冷启动"涉及两个维度：用户等待时间和检索质量。

① 索引构建时间。 对一个 100 个文档（约 5000 个 chunk）的 KB，完整构建流程：文档加载 + 分块（~5s）→ ChromaDB 嵌入（~30s，取决于 GPU/CPU）→ BM25 索引（~0.5s）→ 实体图构建（~2s）。总耗时约 30-40s。ChromaDB 的 embedding 是主要瓶颈。优化方式：使用 GPU 加速 embedding、用 add_texts() 批量而非逐个、或预生成 embedding 文件。

② 检索质量爬坡。 启动时的第一次查询就可能获得高质量结果，因为索引是完整的。但有三点需要注意：第一，BM25 的 IDF 统计在文档数较少时（<20 个）不稳定，但在 100+ 文档后趋于收敛。第二，实体图的共现边在文档少时偏稀疏，需要积累才能形成连通分量。第三，自审循环不依赖冷热状态，始终工作。

③ 渐进式可用策略。 系统的设计允许在索引构建完成前就响应查询——ChromaDB 支持逐 batch 写入，但当前的实现是同步构建。如果用户愿意接受"先建完再用"的模式，30-40s 对首次启动来说是合理的。

LLM 集成 LLM
LLM 输出的 JSON 不稳定怎么办？

DeepSeek（和其他非指令微调的小模型）经常输出格式不稳定的 JSON。我们遇到的主要问题和修复：

① 未引用的 Key——{answer: "..."} 而非 {"answer": "..."}。修复：_repair_json() 中用正则 r'(?<!")(\b[a-zA-Z_]\w*\b)(?=\s*:)' 给 key 加上双引号。

② 字段缺失或字段名不对——输出 "evidence" 而不是 "evidences"。修复：在 _parse_json_response() 中用 setdefault 提供默认值，并做字段名模糊匹配——如果找到 "evidence" 但没找到 "evidences"，自动转换。

③ 字符串内含未转义引号——{"answer": "他说"好的"}"。修复：先用 json.dumps() 尝试解析。失败后回退到逐行扫描修复——检测字符串边界内的未转义引号并加反斜杠转义。

④ 纯文本非 JSON——模型完全没输出 JSON 结构。修复：用 json5 库宽松解析（允许 trailing comma、单引号、无引号 key），如果仍然失败则返回一个包含原始文本的 fallback JSON。

这些修复形成了多层解析管线：先 json.loads → 失败后 repair → 再尝试 → 失败后 json5 宽松解析 → 仍然失败则 fallback。在内部测试中，修复前 JSON 解析成功率约 72%，修复后约 96%。剩余 4% 走 fallback 策略，用户体验基本不受影响。

Self-Review 真的有效吗？验证过程是怎样的？

我们通过对抗测试验证自审效果。具体方法：
③ 构建 50 条测试 query — answer 对，其中 25 条包含故意植入的错误（虚构证据、错误引用、格式异常）。
② 让 LLM 自审打分，记录每条的 score 和是否识别出问题。
③ 统计召回率——有错误的 case 中被自审标记为 score<3 的比例。

结果：召回率约 78%，即 78% 的故意错误被自审捕获。漏网的主要是"事实性错误但表述看起来很合理"的 case 和"对得不明显的数字错误"。相比之下，格式错误和逻辑矛盾的捕获率接近 100%。

假阳性率约 12%——正确的答案被误判为有缺陷（多见于答案较短、证据引用不够"详细"的情况）。为了降低假阳性，我们调整了评分 prompt，明确要求"只有确凿的证据才能给低分"。假阳性虽然增加了一次不必要的重试，但重试后的答案通常没有变差，且第二遍自审通过率 >95%。

结论：自审对格式错误和显式矛盾非常有效，对深层事实幻觉效果有限。它不是一个"幻觉消除器"，而是一个低成本的异常捕获网。要根本解决幻觉问题，需要系统层面的设计——更好的检索质量、引入外部验证源、或使用 RAGAS 等专门评估框架。

DeepSeek 和其他模型（GPT-4、Claude、Qwen）的对比？

在 kb_rag 的典型任务上（意图分类 + 查询重写 + 答案合成 + 自审），我们的非正式对比：

① 意图分类准确率： DeepSeek 约 94% vs GPT-4 约 97% vs Claude 约 96%。差距不大，且 kb_rag 有优先级逻辑（web > clarify > kb > chat）兜底，3% 的误分类不会导致系统性错误。

② 答案质量（主观评分）： GPT-4 ≈ Claude > DeepSeek ≈ Qwen2.5。DeepSeek 在中文生成质量上很好，但在严格的"只基于检索结果"约束上偶尔会加入自己的知识——这在需要高 faithful 的场景中需要注意。

③ JSON 输出规格性： GPT-4 ≈ Claude >> DeepSeek > Qwen。DeepSeek 的 unquoted key 问题明显更频繁，这是为什么我们花了比较多精力做 _repair_json() 的原因。

④ 成本（每百万 token）： DeepSeek 约 $0.5 vs Claude 约 $3 vs GPT-4 约 $10。DeepSeek 是 GPT-4 的 1/20。

选型结论： DeepSeek 的 性价比极高。它将 LLM 集成从"烧钱"变为几乎免费。在 kb_rag 的架构中，LLM 不是唯一的质量关卡（检索质量 + RRF + CrossEncoder + 自审构成了多层保障），LLM 的边际缺陷被系统的其他层补偿。这是"系统设计补偿模型缺陷"的典型例子。

生产化 Production
端到端延迟是多少？每个环节的时间分布？

典型端到端延迟 3-8s。95% 分位约 10s。主要环节分解：

③ 意图分类 + 查询重写 + 实体提取（~0.5-1s）：单次 LLM 调用，与后续检索串行。
② 三路并行检索（~0.8-1.5s）：asyncio.gather 同时发起，延迟由最慢路径决定。ChromaDB 检索通常最快（~200ms），BM25 次之（~500ms），图检索最慢（~1s，取决于 BFS 范围和候选数量）。但这是并行而非串行，所以总检索延迟 ≈ max(dense, BM25, graph) ≈ 1s。
① RRF 融合（~1-5ms）：纯 Python 计算，可忽略。
④ CrossEncoder 精排（~0.8-1.2s）：10 对 query-document 的批处理，取决于 CPU/GPU。
⑤ LLM 答案合成 + 自审（~1-3s）：单次 LLM 调用生成答案 + 自审在同一轮。

优化优先级： 最大瓶颈是 LLM 调用（占总延迟 ~40-60%）。用流式输出（查看首 token 延迟而非全部延迟）能极大改善用户感知。次优是 CrossEncoder（~20%），可用 INT8 量化或 ONNX 降低。

这个系统距离上生产还缺什么？

以生产标准衡量，缺失的关键能力：

① 监控与可观测性。 目前只有 log 日志。上生产需要：请求级别 tracing（OpenTelemetry）、检索质量仪表板（Recall/Latency 面板）、LLM API 错误率和延迟告警、增量更新成功率和异常监控。

② 持久化会话管理。 当前用 InMemorySaver，服务重启后会话丢失。生产环境应迁移到 langgraph-checkpoint-sqlite 或 Postgres 持久化后端的 checkpointer，支持跨重启会话恢复。

③ CI/CD + 测试套件。 目前没有自动化测试。上生产需要：单元测试（每个节点独立测试）、集成测试（完整流水线测试）、回归测试（定期在全量 query 集上运行并对比指标）。

④ 安全管理。 FastAPI 端点目前没有认证。生产环境需要 API key 鉴权、速率限制、用户隔离（多租户 KB 分离）。

⑤ A/B 测试框架。 要持续优化检索参数（chunk size、k、top-K），需要一个 A/B 测试框架来对比不同配置在同一 query 集上的表现。目前是通过手动改参数 + 对比 log 来做，不够系统。

但注意：kb_rag 当前的定位是可工作的架构原型，在功能完整性上已经超过了很多"玩具 RAG"。缺失的是工程化加固层，而非核心能力问题。
